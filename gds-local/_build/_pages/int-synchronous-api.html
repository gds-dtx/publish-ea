    Synchronous API Integration

    Real-time, blocking request/response patterns connecting front-end channels to back-office systems or partner agencies, relying heavily on API Gateways and RESTful standards.

    Synchronous API integration is the most common integration pattern in modern local government architectures. It involves a client (such as a web form, a mobile app, or a partner system) making a request to a server (a back-office system) and waiting (blocking) until it receives a response.

    This pattern ensures that the client immediately knows the outcome of the request (e.g., whether a payment was successful, or if a resident's address was found), making it ideal for interactive user experiences.

    When to use this pattern

    • Real-time lookups: When a user interface needs immediate data, such as a UPRN address lookup or checking council tax balance.
    • Transactional integrity: When the next step in a process cannot continue until the current step is confirmed successful (e.g., payment processing).
    • Partner B2B Integration: When connecting directly to central government services (like DWP APIs) that expect a real-time HTTP request.
    ! Warning Do not use this pattern for long-running processes or moving massive datasets. If a downstream system is slow or unavailable, a synchronous call will block the upstream client, leading to timeouts and a poor user experience. For those scenarios, consider the Asynchronous Messaging pattern.

    Architecture pattern

    The target architecture for synchronous APIs avoids direct point-to-point connections by placing an API Gateway between the consumers and the providers.

    • Consumers: Council website, staff CRM, mobile apps, or external partner systems.
    • API Gateway: The centralised ingress point that handles authentication (OAuth 2.0), rate limiting, threat protection (WAF), and audit logging.
    • Providers: The back-office line of business systems (e.g., Social Care system, Housing system) exposing their native APIs.

    Platforms and tools

    Implementing synchronous API integration effectively requires specific components from the LGAM taxonomy:

    Component Type Examples Role in Architecture
    API Gateway Azure API Management, AWS API Gateway, Kong, Tyk Acts as the secure front door, enforcing policy, rate limits, and routing requests to the correct backend.
    Identity Provider (IdP) Microsoft Entra ID, Auth0, Keycloak Issues the JSON Web Tokens (JWTs) that the API Gateway validates before permitting access.
    Developer Portal Often bundled with the API Gateway A catalogue where developers can discover available APIs, read documentation, and request access keys.

    Target standards

    • REST (Representational State Transfer): The default architectural style for web APIs, using standard HTTP methods (GET, POST, PUT, DELETE).
    • OpenAPI Specification (OAS 3.0+): The industry standard for defining and documenting RESTful APIs. All new council APIs should be documented using OAS to enable auto-generation of SDKs and developer portal documentation.
    • OAuth 2.0 / OpenID Connect (OIDC): The standard for federated identity and API authorisation. Direct basic authentication or static API keys should be avoided in favour of short-lived JWTs.
    • GraphQL: An alternative to REST for complex data retrieval, allowing clients to request exactly the data they need in a single query. Best used for "Experience APIs" sitting in front of mobile or web frontends.

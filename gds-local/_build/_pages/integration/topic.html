    Integration & Interoperability

    Connecting council systems to each other, to partners, and to central government — enabling data and services to flow.

    Integration & Interoperability provides the connective tissue that allows council systems, partner organisations, and central government platforms to exchange data and orchestrate processes. Without deliberate investment in integration capabilities, councils accumulate point-to-point connections that become increasingly fragile and expensive to maintain.

    While the Data Sharing topic defines what data is shared, why, and the information governance surrounding it, this topic covers the physical how: the architectural styles, platforms, and technical patterns required to move that data.

    Why this matters for councils

    • Vendor lock-in and poor APIs: The local gov tech market includes many legacy systems with poor API support, requiring robust integration patterns to extract value.
    • Integration debt: Integration debt accumulates faster than technical debt. Every new system adds point-to-point connections that increase fragility if not managed centrally.
    • RPA as a workaround: Robotic Process Automation (RPA) often exists specifically because proper integration is not available. Councils automate around the gaps rather than solving the underlying connectivity problem.

    Architectural principles

    To minimise integration debt and build a resilient enterprise architecture, councils should adhere to the following principles:

    1. API-first design: Expose all core business capabilities and data through well-documented, secure APIs rather than direct database access.
    2. Avoid point-to-point connections: Use intermediate routing (API Gateways, Event Brokers, or iPaaS) to decouple systems. A point-to-point "spaghetti" architecture creates brittle dependencies that prevent legacy modernisation.
    3. Decouple with events: Prefer asynchronous event-driven integration over synchronous APIs for cross-domain orchestration to improve resilience and scalability.
    4. Secure by design: Never trust the internal network. All APIs and integration endpoints must be authenticated and authorised using standard federated identity protocols (e.g., OAuth 2.0).
    5. Centralised governance, decentralised delivery: Establish common integration standards and patterns centrally, but empower individual delivery teams to build their own integrations using the approved tooling.

    LGAM technology stack at hub level

    Delivering robust integration relies on several core taxonomy nodes within the LGAM Enabling Technology layer.

    LGAM Taxonomy Node Topic Relevance
    API Management & Gateways Platforms and services for designing, publishing, securing, and monitoring APIs, often combined with developer portals for discovery and governance.
    Event-Driven & Streaming Integration Platforms for capturing and distributing real-time event streams (e.g., Kafka, Azure Event Grid), enabling responsive, decoupled architectures.
    Data Integration & Pipelines (ETL/ELT) Tools and processes for extracting, transforming, and loading data between systems, supporting both batch ETL and real-time ELT pipelines.
    Integration Governance & Tooling Secure-by-design integration architecture and governance processes. These establish common integration patterns, standards, and reusable components.
    Message-Oriented Middleware Enterprise Service Buses (ESB) and message brokers that route, transform, and orchestrate interactions between legacy applications.

    Frameworks & guidance

    Relevant standards and procurement vehicles for integration and interoperability:

    • GDS API technical and data standards — How to design, build and operate APIs consistently across the public sector.
    • API Catalogue — Directory of public sector APIs, helping councils discover available government data feeds.
    • Digital and Legacy Application Services (CCS RM6335) — Commercial framework for procuring middleware management, integration platforms, and APIs.
    • Local Digital Declaration — A shared commitment to open standards, insisting on safe and secure technology, and designing systems that communicate with each other.

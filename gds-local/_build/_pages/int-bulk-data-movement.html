    Bulk Data Movement

    Moving large datasets asynchronously on a schedule or trigger, covering modern ETL/ELT data pipelines and legacy Managed File Transfer (MFT) drops.

    While modern digital services prefer real-time, event-driven integration, local government still relies heavily on bulk data movement. This pattern involves extracting large amounts of data from a source system, potentially transforming it, and loading it into a target destination.

    This is primarily used for analytical workloads (feeding a data warehouse) or statutory reporting (nightly extracts to central government), but it is also commonly used as a fallback integration pattern for legacy systems that lack APIs or event streaming capabilities.

    When to use this pattern

    • Data Warehousing & Analytics: Extracting operational data from LOB systems (e.g., Housing, Revenues) to populate a central data lake or warehouse for Business Intelligence.
    • Statutory Returns: Generating and securely transferring large, mandated datasets to central government departments (e.g., DLUHC, DfE) on a scheduled basis.
    • Legacy Integration: When a source system cannot expose an API or publish events, but can produce a nightly CSV dump.
    • System Migrations: The one-off bulk extraction, transformation, and loading of historical records from an old system to a newly procured solution.
    ! Warning Avoid using bulk data movement (like nightly CSV drops) for operational processes that require timely data. If a social worker needs to know if an individual was evicted today, a system relying on a nightly batch job will provide dangerously stale data.

    Architecture pattern

    The architecture for bulk data movement varies based on the modernity of the stack, but generally follows an ETL (Extract, Transform, Load) or ELT (Extract, Load, Transform) approach.

    • Data Sources: SQL Databases, flat files, or API endpoints of operational systems.
    • Orchestrator: A scheduling engine that triggers the data pipelines at specific intervals (e.g., cron jobs, Apache Airflow).
    • Data Pipeline: The compute engine that extracts the data, applies necessary transformations (cleansing, mapping, anonymisation), and loads it to the destination.
    • Destinations: Data lakes, data warehouses, or secure SFTP servers belonging to partner agencies.

    Platforms and tools

    Implementing robust bulk data movement requires enterprise-grade orchestration to handle failures and retries:

    Component Type Examples Role in Architecture
    Cloud Data Pipelines (ETL/ELT) Azure Data Factory, AWS Glue, dbt (data build tool) Modern, scalable services for authoring, scheduling, and monitoring complex data transformations.
    Data Orchestrators Apache Airflow, Prefect, Dagster Platform for authoring, scheduling, and monitoring data pipelines as Directed Acyclic Graphs (DAGs).
    Managed File Transfer (MFT) Globalscape, MoveIT, Azure SFTP Secure, audited transfer of flat files (CSV, XML) between external partners or legacy systems.

    Target standards

    • Data Formats: Parquet or Delta Lake for modern analytical storage; CSV or JSON-L for legacy flat file transfers.
    • Encryption: Data transferred over the internet must use TLS 1.2+ in transit. PGP encryption should be applied to sensitive file payloads (at rest) before transmission via SFTP.
    • Idempotency: Data pipelines should be designed idempotently, meaning if a nightly job fails halfway through and is restarted, it does not duplicate data in the target system.

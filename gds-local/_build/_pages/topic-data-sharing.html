    Data Sharing

    A conceptual model and common patterns for purposeful data sharing across the public sector.

    Local government holds and processes vast amounts of personal and non-personal data. Sharing this data effectively—while remaining lawful, proportionate, and secure—is critical to delivering joined-up services, protecting vulnerable people, and reducing administrative burdens on citizens.

    This topic hub draws on the principles established in the iStandUK Role-Based Data Sharing White Paper to provide a domain-agnostic, configurable conceptual reference for data architects and information governance leads in local government.

    Data sharing patterns

    Because technical implementation varies drastically depending on the actors and purpose, data sharing is best understood by mapping it to specific sharing patterns. Select a pattern below to view detailed technical lifecycles, target architectures, and standards.

    Statutory cross-government sharing

    Sharing data between local and central government under specific legal gateways (e.g. DEA 2017 Part 5), relying on APIs, established national data standards, and, where mandated by legacy external systems, scheduled bulk transfers.

    Inter-authority and partner sharing

    Data exchanged between local public services (e.g. Councils, NHS trusts, Police) usually driven by public task or substantial public interest, using data sharing agreements and attribute-based access controls.

    Empowering residents to hold and share their own verified attributes (e.g., through digital wallets or verifiable credentials) to prove eligibility or status.

    Publishing non-personal or anonymised datasets for public consumption, transparency, and re-use by third-party innovators using open data standards.

    Architectural principles

    Local government sharing arrangements should conform to core principles, ensuring security, interoperability, and legal compliance:

    1. Legal basis before integration: Technical integration must not precede governance. The lawful basis must be established and documented before technical pipelines are built.
    2. Catalogue as a first-class artefact: Data cannot be shared effectively if it cannot be discovered. Data schemas and assets must be catalogued using established standards (e.g., DCAT v3, OpenAPI 3).
    3. Derived data to honour minimisation: Architectures must support querying and returning derived data (e.g., isOver65: true) rather than sharing full raw records, strictly fulfilling UK GDPR data minimisation principles.
    4. Standards-based identity and audit: Point-to-point IP whitelisting and hardcoded service accounts are insufficient. Federated identity (e.g., OAuth 2.0 / OIDC) must be used to ensure granular, per-request auditability tied to specific users and roles.
    5. Honour the data subject: Design architectures that respect the data subject. Allow citizens visibility into how their data is shared and processed, and consider citizen-mediated sharing patterns (e.g., verifiable credentials) where appropriate.

    LGAM technology stack at hub level

    Delivering robust data sharing architectures relies on several core taxonomy nodes within the LGAM. These capabilities must be provisioned and configured to support the sharing patterns detailed above.

    LGAM Taxonomy Node Topic Relevance
    Identity and access management (IAM) Provides the federated trust mechanisms (OAuth 2.0, OIDC) needed to authenticate both workforce professionals and citizens, and securely authorise machine-to-machine B2B integration.
    Partner (B2B) Integration Specialised integration capabilities to connect systems across organisational boundaries, handling varying protocols and security requirements of partner agencies.
    Data Catalogue & Metadata The foundational capability for making data assets discoverable across and outside the organisation, enabling stakeholders to understand what data exists and how it can be accessed.
    API Management & Gateways Centralised points of ingress and egress for sharing data via APIs. They enforce rate limits, authenticate consumers, and provide the critical audit logging necessary for statutory compliance.
    Security Operations (SIEM/SOC) Centralised collection and monitoring of immutable audit logs generated by integration gateways and shared records to ensure data is accessed legally and proportionately.

    The conceptual model

    For personal data to be shared safely and lawfully, the data controller needs assurance. As defined by the iStandUK White Paper, the core requirement is:

    A user is able to assert that they are empowered by their organisation to act in a role that has a right to a set of data items for a purpose, and agrees to the terms by which the data is to be used and handled.

    These seven elements provide a structured way to configure and describe any data sharing arrangement.

    Element Definition Reference Frameworks
    Purpose The objective for which data can be shared. Digital Economy Act 2017 Part 5; Documented public task under UK GDPR.
    Data Items The specific subset of data required to support the purpose. Data Catalogues using DCAT v3; UK Cross-Government Metadata Exchange Model.
    Right The legal basis for disclosure. UK GDPR lawful bases; DEA 2017 gateways.
    Organisation The entities that have the right to the data. Specified persons in legislation, or statutory identifiers.
    Role The role a user must hold to exercise the right. Role-Based Access Control (RBAC); Attribute-Based Access Control (ABAC).
    Assertion The required strength of identity verification. OIDC/OAuth 2.0; UK digital identity and attributes trust framework (DUAA 2025).
    Terms Information governance undertakings. Data Sharing Agreements; DPIAs; ICO code of practice.

    Target data and metadata standards

    Data sharing requires adherence to established data standards to ensure interoperability. Key standards include:

    • DCAT v3: The W3C Data Catalog Vocabulary is the standard for publishing metadata about datasets.
    • UK Cross-Government Metadata Exchange Model: The UK extension to DCAT to enable cross-government cataloguing.
    • OpenAPI Specification 3: The standard for describing and documenting RESTful APIs to ensure programmatic interoperability.
    • GOV.UK Guidance: Identify and record critical data assets: National guidance on data cataloguing.

    Additionally, for sharing data concerning vulnerable individuals, SAVVI provides a canonical operational example and data standard for vulnerability use cases.

    Legal context

    Public sector data sharing operates within a strict legislative environment. Key instruments include:

    • UK GDPR & Data Protection Act 2018: Establishes the comprehensive personal data protection regime, requiring a lawful basis (often "public task" Article 6(1)(e)) and adherence to principles like data minimisation and transparency.
    • Digital Economy Act (DEA) 2017, Part 5: Provides statutory gateways for specified public bodies to share data for public service delivery, fraud prevention, debt management, research, and statistics.
    • Data (Use and Access) Act (DUAA) 2025: Amends the UK GDPR to clarify legitimate interests and rules around automated decision making, while establishing a statutory footing for Digital Verification Services.

    Adjacent frameworks

    Enterprise Architects should align their data sharing capabilities with established cross-government frameworks:

    • The Technology Code of Practice – A set of criteria to help government design, build and buy technology.
    • NCSC Cyber Assessment Framework (CAF) – Guidance for assessing the extent to which cyber risks are being managed.
    • The Algorithmic Transparency Recording Standard (ATRS) – A framework to help public sector organisations be transparent about algorithmic tools.
    • The Data Ethics Framework – Principles to guide the design of appropriate data use in the public sector.
    • SAVVI – Scalable Approach to Vulnerability via Interoperability (a canonical operational example and data standard for vulnerability use cases).
    • The Local Digital Declaration – A shared ambition for the future of local public services written by a collective of local authorities and sector bodies.

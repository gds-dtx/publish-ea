    Asynchronous Messaging

    Real-time, non-blocking, decoupled patterns (publish/subscribe, fire-and-forget) using event brokers to notify downstream systems of state changes.

    Asynchronous messaging allows systems to communicate without having to wait for a response from the receiving system. When a system performs an action (e.g., an address is updated in the CRM), it publishes a message or "event" to a central broker. Other systems that care about this event (e.g., Council Tax, Waste Management) can subscribe to the broker and process the update at their own pace.

    This architectural style fundamentally decouples the publisher from the subscriber, meaning the publisher doesn't need to know who is listening, and if a subscriber system goes offline, the messages are safely queued until it returns.

    When to use this pattern

    • Cross-domain updates: "Tell Us Once" style scenarios. If a resident updates their name in the citizen portal, this event can be broadcast asynchronously to update the housing, revenues, and social care systems without locking up the portal UI.
    • Resilience against downtime: If a legacy back-office system frequently goes offline for nightly backups, upstream web forms can drop messages onto a queue to be processed when the system comes back online, preventing data loss.
    • Orchestrating long-running processes: Such as document generation or complex eligibility calculations that take too long to hold open a synchronous HTTP connection.
    Event-driven vs Message-driven: While similar, an event usually just announces that something happened (e.g., "AddressUpdated"), whereas a message (command) tells another system to do something specific (e.g., "PrintLetter"). Both use asynchronous brokers, but events provide greater decoupling.

    Architecture pattern

    The core of an asynchronous architecture is the Event Broker or Message Queue.

    • Publishers (Producers): The system where the state change occurs. It formats the event (often in JSON) and pushes it to a topic on the broker.
    • Event Broker: The intermediary platform. It receives the event, stores it durably, and routes it to any registered subscribers.
    • Subscribers (Consumers): Systems that listen to specific topics. They pull the event from the broker (or are pushed via Webhook) and update their own local data stores accordingly.

    Platforms and tools

    Implementing asynchronous messaging relies on robust middleware:

    Component Type Examples Role in Architecture
    Message Queues Azure Service Bus, AWS SQS, RabbitMQ Point-to-point asynchronous delivery. Ensures a command is delivered exactly once to a specific worker.
    Event Streaming Platforms Apache Kafka, Azure Event Hubs, AWS Kinesis High-throughput publish/subscribe platforms where multiple consumers can read the same stream of events.
    Webhooks GOV.UK Notify receipts, Stripe events A lightweight form of asynchronous push integration over HTTP. Usually requires the subscriber to expose a public endpoint.

    Target standards

    • AsyncAPI: The industry standard for documenting event-driven architectures. Similar to how OpenAPI documents REST APIs, AsyncAPI documents the channels, messages, and payloads of a message broker.
    • CloudEvents: A specification by the Cloud Native Computing Foundation (CNCF) for describing event data in common formats to provide interoperability across services, platforms, and systems.
    • AMQP (Advanced Message Queuing Protocol): An open standard application layer protocol for message-oriented middleware.

    Integration & Interoperability

    About this domain

    Connecting council systems to each other, to partners, and to central government — enabling data and services to flow. Covers Application Programming Interface (API) management, integration platforms, data pipelines, national service connectivity, inter-authority data sharing, and standards compliance.

    Integration & Interoperability sits in the Enabling Technology layer of the Local Government Architecture Model. It provides the connective tissue that allows council systems, partner organisations, and central government platforms to exchange data and orchestrate processes. Without deliberate investment in integration capabilities, councils accumulate point-to-point connections that become increasingly fragile and expensive to maintain.

    Why this matters for councils

    Understanding the local government context for integration and interoperability decisions:

    • Vendor lock-in and poor APIs — the local gov tech market is dominated by a small number of vendors (Civica, Capita, NEC/Northgate, TechnologyOne, Liquidlogic), many of whose legacy systems have poor API support.
    • RPA as a workaround — RPA exists specifically because proper integration is not available. Councils automate around the gaps rather than solving the underlying connectivity problem.
    • Two-tier data sharing — in two-tier areas, county and district councils need to share data between their respective systems (e.g., social care data at county linking to housing data at district).
    • Central government expectations — central government increasingly expects councils to integrate with national platforms (Tell Us Once, One Login, Planning Portal, Department for Work and Pensions (DWP)/His Majesty's Revenue and Customs (HMRC) data feeds).
    • Data silos — data silos are the single biggest technology pain point. Citizens repeat information across council services because systems do not talk to each other.
    • Integration debt — integration debt accumulates faster than technical debt in councils. Every new system adds point-to-point connections that increase fragility.

    Key capabilities

    The core integration and interoperability capabilities that councils need to connect their technology estate.

    API Management

    Publishing, securing, versioning, and governing APIs for internal and external consumers. Councils increasingly need to expose their data and accept inbound submissions via APIs (e.g., planning submission API, waste collection API).

    Integration Platform

    Middleware or Integration Platform as a Service (iPaaS) for connecting disparate council systems. Handles transformation, orchestration, and routing. Examples: MuleSoft, Azure Integration Services, Boomi. Reduces point-to-point spaghetti.

    Data Pipelines

    Moving and transforming data between systems. Extract, Transform, Load (ETL)/Extract, Load, Transform (ELT) for batch and near-real-time data flows. Critical for feeding data warehouses, statutory returns, and cross-service analytics.

    National Service Connectivity

    Integrating with central government platforms: DWP (benefits verification), HMRC (council tax), Tell Us Once (life events), Planning Portal (applications). Each has its own API patterns and onboarding requirements.

    Inter-Authority Data Sharing

    Secure, governed exchange of data with partner organisations: other councils, NHS, police, housing associations, voluntary sector. Particularly critical in two-tier areas and for safeguarding.

    Standards & Protocols

    Compliance with interchange standards: Open311, UPRN (Unique Property Reference Number), LocalGov Drupal schemas, GDS API design patterns, Open Referral UK (community services), and emerging local gov data standards.

    What to consider

    Key questions when assessing integration and interoperability capabilities:

    • How many point-to-point integrations do you have, and do you have visibility of them all?
    • Do your core vendors offer modern APIs, or are you relying on file-based transfers and RPA?
    • Can you integrate with central government services today?
    • Do you have an integration platform, or does each system team build their own connections?
    • How do you share data with partner organisations securely — and is it governed?
    • Are you using UPRN as a common key across your property-related systems?

    Available platforms & tools

    GDS / Public Sector

    API Catalogue — directory of public sector APIs, helping councils discover available government data feeds and services.

    GDS API Design Guidance — patterns for building government APIs, applicable to councils publishing their own APIs.

    Open Referral UK — standard for community services data, enabling consistent directory publishing across councils.

    Commercial (commonly used by councils)

    • Microsoft Azure Integration Services — Logic Apps, Service Bus, API Management. The natural choice for Microsoft-ecosystem councils.
    • MuleSoft Anypoint Platform — enterprise iPaaS for API-led connectivity and system integration.
    • Dell Boomi — cloud-native integration platform for connecting Software as a Service (SaaS) and on-premises systems.
    • Jitterbit — integration platform common in Salesforce-based councils.
    • Apache NiFi — open source data pipeline automation for batch and streaming data flows.
    • Local authority vendor APIs — Civica Open API, NEC Connect, Capita APIs (varying quality and completeness).

    Standards & guidance

    Relevant standards and guidance for integration and interoperability decisions:

    • GDS API technical and data standards — how to design, build and operate APIs consistently
    • GDS API Design Guidance — detailed patterns for RESTful API design
    • Open311 — standard for civic issue reporting (used by FixMyStreet etc.)
    • UPRN standard — unique property reference number for address matching across systems
    • Local Digital Declaration — commitment to open standards and interoperability

    Commercial frameworks

    Crown Commercial Service (CCS) agreements relevant to integration and interoperability procurement:

    • Digital and Legacy Application Services — middleware management, integration, APIs, API platforms and Enterprise Service Bus (ESB)s
    • Technology Services — technology service agreements including integration consultancy

    Connections to service delivery

    Integration & Interoperability enables the entire LGAM service delivery stack:

    • Workflow — case management relies on integration to pull and push data across systems (e.g., referrals flowing from web form to social care system)
    • Forms — online forms need to write submissions to multiple back-end systems
    • Payments — payment reconciliation flows between payment platform and income management
    • All IT Systems — every system needs to connect; the integration platform is the connective tissue
    • Business Areas — cross-service processes (e.g., housing + benefits + council tax) depend on system integration

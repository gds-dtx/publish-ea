      Development Control

      The technical architecture supporting the lifecycle of an individual planning application—from the citizen's initial submission to the council's final legal determination.

      Because this service is highly transactional and public-facing, its architecture must handle high volumes of web traffic, complex spatial rule evaluations, and heavy unstructured document processing. It represents the bulk of a council's planning technology expenditure.

      The LGAM technology stack

      Delivering this journey requires orchestrating multiple components across the LGAM taxonomy. An API-first architecture decouples these capabilities rather than purchasing them as a single monolithic system.

      Core LGAM capabilities required for Development Control
      LGAM Taxonomy Node Domain Specific Implementation
      Forms Dynamic, spatial-aware frontends (e.g., PlanX) that use "rules-as-code" to ask contextual questions based on the property's location (e.g., Conservation Areas).
      Workflow The core back-office engine (e.g., BOPS) managing statutory timelines, tasks, consultations, and the officer determination process.
      Document and records management Secure storage for massive volumes of unstructured data: architectural plans, CAD files, public comments, and decision notices.
      Geospatial data The spatial database providing UPRN lookups, mapping polygons for application boundaries, and intersecting proposals with local plan constraints.
      API management and gateways RESTful endpoints connecting the public submission portal to the back-office, and integrating with statutory consultees (e.g., Environment Agency).
      Identity and access management (IAM) Federated Identity (OIDC/SAML) specifically for high-frequency planning agents, allowing them a single sign-on view of all their active applications.

      The technical lifecycle

      This section outlines the business process at each phase of a planning application. Expand each phase to compare the neutral baseline architecture against target architectural models.

      1 Pre-application & Submission

      Phase 1 Architecture

      • Business Process
      • Legacy Anti-patterns
      • Target Architecture

      The process: The intake of data from citizens, architects, and planning agents. The council securely captures the applicant's intent, property details, supporting documents, and the specific geographic boundaries of the proposed development.

      Required LGAM Taxonomy Nodes:

      Forms Identity and access management (IAM) Document and records management
      Anti-pattern: Generic E-Forms to PDF

      Using generic council web forms that capture free-text input and immediately flatten the data into an unstructured PDF. This strips the submission of its machine-readable data on day one.

      Anti-pattern: Free-text address matching

      Allowing applicants to manually type their address rather than forcing a UPRN lookup against the corporate gazetteer, resulting in failed spatial linkages downstream.

      Best Practice: Composable rules-as-code frontend

      Modern architectures utilise rules-as-code engines (e.g., PlanX) instead of static forms. These systems branch intelligently, interacting with APIs mid-journey to calculate complex fees and fetch spatial constraints (like Conservation Areas) before the user even submits.

      The resulting payload is standardised JSON aligned to the MHCLG schema, meaning the back-office workflow engine receives pre-structured, validated data rather than a generic PDF.

      Real-world impact
      • Reduced failure demand

        Catching errors and calculating exact fees dynamically prevents invalid applications from ever entering the system, eliminating hours of back-and-forth emails for officers.

      • Better citizen UX

        Applicants get immediate clarity on constraints (e.g., "You are in a flood zone") before paying fees.

      2 Validation & Triage

      Phase 2 Architecture

      • Business Process
      • Legacy Anti-patterns
      • Target Architecture

      The process: Ensuring the submission is legally complete, geographically sound, and financially settled before officer assessment begins. This involves checking architectural plans, validating ownership certificates, calculating fees, and drawing the site boundary polygons into the corporate GIS system.

      Required LGAM Taxonomy Nodes:

      Workflow Payments Geospatial data
      Anti-pattern: Swivel-chair integration

      Validation officers forced to look at a PDF plan on one screen while manually clicking to redraw identical redline boundaries into a separate GIS database.

      Best Practice: Machine Learning extraction

      Validation is traditionally a major bottleneck caused by unstructured data. To reduce manual data entry from legacy formats, councils are introducing computer vision and LLMs (such as the government's MHCLG / i.AI Extract tool) as a processing middleware.

      These models parse inbound PDFs, isolate the redline boundary pixels, convert them into structured GeoJSON, and extract form intent into standard schema variables.

      Real-world impact
      • Operational efficiency

        Drastically reduces manual transcription and redrawing time, clearing validation backlogs faster and lowering processing costs.

      • Resource allocation

        Frees highly-skilled validation officers from administrative data-entry, allowing them to focus entirely on assessing complex edge-cases and legal compliance.

      3 Consultation & Assessment

      Phase 3 Architecture

      • Business Process
      • Legacy Anti-patterns
      • Target Architecture

      The process: The core officer workflow. Managing the statutory public comment period, processing neighbour letters, performing spatial assessments, and exchanging information with external statutory bodies (e.g., Environment Agency, Highways).

      Required LGAM Taxonomy Nodes:

      Workflow Websites Partner (B2B) integration
      Anti-pattern: Contended Databases

      Pointing high-traffic public registers directly at the synchronous back-office workflow database. A controversial planning application can cause public traffic spikes that slow down or crash the internal system for officers.

      Anti-pattern: Manual Consultee PDFs

      Generating consultation requests as PDF letters and emailing them manually to statutory bodies, rather than passing the structured application payload via an API.

      Best Practice: Decoupled edge registers and API integrations

      A core architectural risk in this phase is public traffic degrading internal system performance (e.g., during a controversial housing development consultation). Best practice dictates that the public-facing register utilises read-replicas, asynchronous edge-caching, and CDNs, fully decoupling public traffic from the synchronous database used by officers.

      For statutory consultations, manual PDF letter generation is replaced by Partner (B2B) integrations, routing data directly to external bodies via API Gateways.

      Real-world impact
      • System resilience

        Protects core council infrastructure from Denial of Service (DoS) conditions during high-profile planning disputes, ensuring back-office staff can keep working.

      • Statutory compliance

        Automated API routing to consultees creates a watertight, auditable log of correspondence, reducing the risk of legal challenges over missed consultation deadlines.

      4 Determination & Post-Decision

      Phase 4 Architecture

      • Business Process
      • Legacy Anti-patterns
      • Target Architecture

      The process: Issuing the final legal decision, recording conditions, preparing bundles for potential appeals, and publishing the final outcome data to central government repositories.

      Required LGAM Taxonomy Nodes:

      Document and records management Event-driven and streaming integration
      Anti-pattern: Manual File Bundling

      When an appeal is lodged, officers manually download dozens of PDFs, compile them into a ZIP file, and email them to the Planning Inspectorate (PINS), creating massive administrative overhead.

      Best Practice: Event-driven API publication

      To reduce administrative overhead, document assembly automation should programmatically generate the Decision Notice by pulling structured data (conditions, constraints) from the database.

      Simultaneously, target architectures utilize Event Brokers. Upon determination, a status event is fired which pushes the structured decision data automatically to planning.data.gov.uk and, in the event of an appeal, transfers the entire case metadata payload securely to PINS via REST API, bypassing manual ZIP creation entirely.

      Real-world impact
      • Administrative savings

        Eliminates the manual process of building appeal bundles, a historically massive drain on officer time.

      • Data transparency

        Seamless, automated compliance with central government open data mandates without requiring manual CSV exports or ETL maintenance.

      Target data models

      When procuring or configuring systems for Development Control, the architecture must align with the national data schemas maintained by MHCLG on planning.data.gov.uk. Treating these schemas as your target data model ensures automated interoperability with central government and neighboring authorities.

      Dataset Schema Architectural Purpose
      Planning applications The core transactional record. Target architectures must natively emit changes to application status (e.g., 'Registered', 'Determined') using this exact schema via an Event Broker.
      Conservation areas / Tree preservation orders Spatial constraints. Instead of copying these as static shapefiles, the rules-engine frontend should query these schemas via API to dynamically determine validation requirements.
      Developer contributions (Section 106) Financial obligations linked to planning permissions. Requires integration between the Workflow DB and the council's Financial capabilities using this structured data model.

      Platforms & tools

      The market for Development Control systems encompasses both comprehensive enterprise suites and an emerging ecosystem of interoperable, API-first tools.

      Established Enterprise Suites

      These platforms provide comprehensive, end-to-end functionality, typically bundling the public register, workflow, document management, and spatial tools into a unified environment.

      • Idox Uniform — A comprehensive back-office suite, often paired with 'Public Access' for the citizen portal.
      • NEC Assure — A cloud-based regulatory services platform covering planning, building control, and land charges.
      • DEF Software — Modular, Microsoft-aligned planning and regulatory software.
      • Arcus Global — Salesforce-based regulatory services platform.

      Composable Ecosystem

      Tools often emerging from initiatives like the Open Digital Planning (ODP) project. These are decoupled components designed to connect via open APIs.

      • PlanX — A rules-as-code frontend engine for intelligent application submission and self-service validation.
      • Back Office Planning System (BOPS) — A targeted workflow engine specifically designed for officer assessment and determination.
      • i.AI Extract — Government-built AI tool for extracting structured data from unstructured PDFs.

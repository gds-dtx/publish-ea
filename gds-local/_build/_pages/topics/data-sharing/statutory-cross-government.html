    Statutory cross-government sharing

    The technical and governance architecture for sharing data between central and local government under explicit statutory gateways.

    This pattern typically relies on the Digital Economy Act 2017 (Part 5) or specific domain legislation (such as the Care Act or Children Act). These arrangements are characterised by high volumes, strict legal definitions of permitted purpose, and increasing reliance on secure B2B APIs rather than legacy bulk file transfers.

    LGAM technology stack at pattern level

    Delivering this pattern requires orchestration of integration and security capabilities within the LGAM taxonomy.

    LGAM Taxonomy Node Pattern Specific Implementation
    API management and gateways Centralised ingress/egress points that enforce rate limiting, monitor usage, and provide audit logging for statutory compliance.
    Data integration (ETL/ELT) Pipelines that ingest national data feeds, transform them to match local schemas, and load them into local operational stores or data platforms.
    Identity and access management (IAM) Depending on the use case, either federated User-Level tokens (e.g. OAuth 2.0 Authorization Code flow) for interactive queries requiring individual audit trails, or Machine-to-Machine authentication (e.g. OAuth 2.0 Client Credentials flow) for automated, high-volume backend pipelines.
    Data cataloguing Maintaining metadata records of data assets shared with central government to satisfy cross-government critical data asset requirements.

    The technical and governance lifecycle

    Establishing and running a statutory cross-government data share involves several phases.

    1 Information Governance & Agreement

    • Target Architecture
    • Legacy Anti-patterns

    Target: Configuration Driven Mappings

    The Data Sharing Agreement (DSA) explicitly mandates the technical Assertion (e.g. OAuth 2.0 federation) and Role requirements. Data controllers evaluate the sharing arrangement against the seven conceptual elements to ensure the legal basis maps directly to technical controls.

    Anti-pattern: Build first, govern later

    Building technical integration pipelines before establishing the lawful basis. Information Governance must be established before technical builds commence.

    Anti-pattern: Point-to-Point Justification

    Drafting bespoke DPIAs and DSAs from scratch for every minor data share, rather than leveraging scalable frameworks to standardise governance.

    2 Technical Discovery & Handshake

    • Target Architecture
    • Legacy Anti-patterns

    Target: Directory of Standardised API Services

    Data schemas are published as machine-readable OpenAPI 3 specifications, and the data assets themselves are catalogued using DCAT v3 standards. Federation is established by registering the Consumer's Identity Provider with the Controller's API Gateway.

    Anti-pattern: Hardcoded Service Accounts

    Using static database credentials or API keys shared via email, which are never rotated and lack identity federation.

    Anti-pattern: IP Whitelisting Insufficiency

    Relying solely on IP whitelisting to secure the exchange, which provides no visibility into the individual user or their asserted role.

    3 Execution & Audit

    • Target Architecture
    • Legacy Anti-patterns

    Target: Derived Data with Immutable Logs

    The Consumer queries the API passing an OAuth 2.0 access token in a cryptographically signed format (e.g. a JWT). For interactive queries, this token explicitly identifies the individual user; for automated pipelines, it identifies the consuming service. The Controller evaluates the token's signature mathematically without needing to query the issuing IdP, queries the internal Core DB, and returns only the derived data needed to satisfy the purpose (e.g. isEligible: true). Every access event is written to an Immutable Audit Log, allowing the Controller to respond easily to Subject Access Requests (SARs).

    Anti-pattern: Massive SFTP Data Dumps

    Relying on batch-generated CSV files containing thousands of raw records, uploaded to a secure FTP server. This over-shares personal data, violates data minimisation principles, and relies on manual ingestion processes.

    Target data and metadata standards

    Statutory pipelines should adhere to strict, machine-readable standards to ensure interoperability and compliance:

    • OpenAPI Specification 3: The GOV.UK recommended standard for describing RESTful APIs. It provides a machine-readable contract for integration.
    • DCAT v3 / UK Cross-Government Metadata Exchange Model: Data assets exchanged under statutory gateways must be properly catalogued using these standard vocabularies.

    Platforms and tools

    Implementing this pattern typically involves enterprise-grade integration platforms:

    • API Gateways: Tools like Azure API Management, AWS API Gateway, or Kong are used to enforce rate-limiting, authentication, and logging at the boundary.
    • Integration Platforms (iPaaS): Platforms like Dell Boomi, MuleSoft, or Azure Logic Apps orchestrate the connection to legacy line-of-business systems and handle data transformation.
    • Identity Providers (IdP): Enterprise IdPs (like Entra ID or Okta) issue and validate the federated OAuth 2.0 or OIDC tokens securing the exchanges.

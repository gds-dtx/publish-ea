    Inter-authority and partner sharing

    The technical and governance architecture for sharing data horizontally across local multi-agency partnerships.

    This pattern covers data exchange between local public services—such as upper and lower-tier councils, NHS trusts, police forces, and fire services. Common use cases include Multi-Agency Safeguarding Hubs (MASH), health and social care integration, and vulnerability identification. It typically relies on the UK GDPR "public task" or "substantial public interest" lawful bases, governed by multiparty Data Sharing Agreements (DSAs).

    LGAM technology stack at pattern level

    Delivering this pattern requires complex orchestration of identity, security, and master data management.

    LGAM Taxonomy Node Pattern Specific Implementation
    Identity and access management (IAM) Federated identity allowing a professional from one agency (e.g., an NHS nurse) to securely authenticate into a shared portal or access specific data attributes based on their professional role (RBAC/ABAC).
    Master Data Management (MDM) Probabilistic matching engines used to resolve identities across systems that lack a shared primary key (e.g., linking a council housing record to an NHS patient record using name, DOB, and address history).
    API management and event brokers Pub/sub messaging systems that allow one agency to subscribe to critical events published by another (e.g., alerting social care when a known vulnerable adult is admitted to A&E).
    Document and records management Shared care records or multi-agency data trusts providing a unified, read-only view of a citizen's interactions across the local partnership.
    Audit and accountability Immutable audit logging capturing every data access event tied to the specific user's identity, stored centrally to support multi-agency investigations and Subject Access Requests (SARs).

    The technical and governance lifecycle

    Establishing a multi-agency sharing agreement requires careful balancing of technical integration and privacy controls.

    1 Information Governance

    • Business Process
    • Best Practice

    The process: Negotiating and signing a multi-party Data Sharing Agreement (DSA) that defines the purpose, the roles allowed access, and the specific data attributes to be shared.

    Best Practice: Derived Data

    As outlined in the iStandUK white paper, partners should default to sharing derived data rather than raw data to satisfy data minimisation principles. For example, instead of sharing a complete medical record, an NHS system might expose a binary flag: "Unfit to drive" or "Requires ground-floor accommodation".

    2 Identity Resolution

    • Target Architecture
    • Anti-patterns

    When bringing together data from the NHS (using NHS Number), the Council (using UPRN/internal ID), and Police (using PNC), a Master Data Management (MDM) solution uses probabilistic matching algorithms on name variations, dates of birth, and historical addresses to create a "Golden Record" without permanently merging the source databases.

    Anti-pattern: Deterministic Exact Matching

    Relying on exact string matches for First Name, Last Name, and Date of Birth across databases. This approach fails to account for typos, nicknames, name changes, or slight date errors, resulting in high rates of missed linkages (false negatives) for vulnerable people.

    Target data and metadata standards

    Effective multi-agency sharing relies on common terminology and structured metadata:

    • DCAT v3 / UK Cross-Government Metadata Exchange Model: For cataloguing the data assets available across the partnership.
    • Domain Standards: e.g., FHIR (Fast Healthcare Interoperability Resources) for NHS integration, or specific Police/Home Office schema formats.

    Platforms and tools

    Implementing this pattern usually relies on a mix of identity and integration engines:

    • Master Data Management (MDM): Tools like IBM InfoSphere, Informatica, or specialized public sector MDMs for identity resolution and golden record creation.
    • Event Brokers: Event streaming platforms like Apache Kafka, Azure Event Grid, or RabbitMQ to route critical safeguarding events in real-time.
    • Federated Identity Providers: Platforms that support SSO and attribute-sharing across different domains (e.g., Entra ID, Auth0).

      Planning and Development

      The processing of planning applications, enforcement of building regulations, and creation of local spatial plans.

      Planning and Development represents one of the most legally complex, highly scrutinised, and data-intensive functions of local government. Systems in this domain must handle massive volumes of unstructured data (architectural plans, public objections) whilst strictly enforcing spatial geometries and statutory deadlines.

      Domain architectural principles

      While the value streams above have specific needs, Enterprise Architects should enforce these core principles across all technology procured or built for the Planning and Development domain.

      1. The Spatial Golden Thread (UPRNs/USRNs)

      Every planning application, building control certificate, and enforcement notice should be fundamentally anchored to a Unique Property Reference Number (UPRN) or Unique Street Reference Number (USRN). Systems that rely on free-text addresses or proprietary gazetteers should be phased out, as they break interoperability with corporate systems (like Waste and Revenues).

      2. Separation of Document Storage from Transactional State

      Planning generates terabytes of unstructured data (PDFs, CAD files, photographs). Target architectures decouple this heavy document storage from the primary relational database managing the workflow state. This prevents database bloat and ensures the core workflow engine remains highly performant.

      3. API-First Statutory Integration

      Interactions with statutory bodies (e.g., The Environment Agency, Historic England, the Planning Inspectorate) should transition from manual PDF exchanges to structured B2B API integrations. When procuring new back-office systems, architects should prioritise native support for external API webhooks to handle automated status updates.

      The target data model

      The Ministry of Housing, Communities and Local Government (MHCLG) maintains national data standards for the planning sector at planning.data.gov.uk. Enterprise Architects should view these not merely as reporting requirements, but as the target schema for their own internal systems.

      Best practice for procurement

      When drafting requirements, councils should specify that new planning software can natively export and ingest data matching these national schemas. Aligning to these standards early avoids the need to build and maintain complex, bespoke ETL (Extract, Transform, Load) pipelines later.

      Data Schema Description & Use Case
      Planning applications Standardised lifecycle records of applications, including site boundaries, proposed usage, and determination outcomes.
      Local plans Machine-readable versions of spatial planning policies, enabling automated rules-engines to check constraints before submission.
      Developer contributions Section 106 agreements and Community Infrastructure Levy (CIL) data, crucial for integrating planning with corporate financial systems.
      Brownfield land Registers of previously developed land suitable for housing, demonstrating open spatial data publishing.
